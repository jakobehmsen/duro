Overview
Duro is an attempt to make a runtime for persistent processes. The contents of this project is freely available on GitHub under an MIT license.
Motivation
For most runtimes, processes are transient which means, when they are restarted, they are conceptually dependent on external persistent computing elements to recap their whereabouts at their last exit.
What if this wasn't the default case for processes? What if, instead, processes, after they have been stopped, simply always continue from where they left of?
Possibilities
Many applications have essentiel parts that must persist across process incarnations. The most common contemporary solution is to use external components, such as files (flatfile, XML, CSV, ...) or databases (SQL, NoSQL, ...). Integrating with these external components can be challenging and very time consuming depending on the degree of impedance mismatch. Further, this may implicate issue with down-scaling, i.e. starting out simple is difficult.
It is likely that a significant part of these applications could use persistent processes for persisting these essential parts. If so, then probably no (or very little) effort is required to support the persistence of these parts.
Challenges
The idea of persistent processes may implicate a naive implementation, where the size of the processes increases very fast. Some sort of maintenance of the persistent processes are likely necessary and challenging to implement.
Since it is not just executables which are created but processes, updating application behavior cannot be done by simply replacing an executable since that'll destroy the existing evolution of the previously compiled process. Therefore, a seperation between process program and state could be valuable.
Crystalization
The current goal is to implement a frontend, inspired by, among others, ECMAScript and Smalltalk, for the runtime simply named Duro. Duro is to have the same role as Java has for the JVM. With a twist. Where Java is compiled to programs, from which processes are derived from - Duro is compiled directly to processes which are simply resumed each time they are given to the runtime. The initially compiled processes are called genesis processes. Basically, a process is realized by combining a program with an interaction history. An interaction history represent the state of a process. This way, the program is not entangled with the process - and thus, the program part of a process can be replaced without affecting the state of the process.
